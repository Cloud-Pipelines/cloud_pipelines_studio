Custom Service Accounts

Oasis is designed for collaboration and chared use across teams. All workloads (pipeline task executions) normally use same service account (oasis-service-account@shopify-skypilot-clusters.iam.gserviceaccount.com). However, some teams wish to use custom service account to that restricted data securely. This guide demonstrates how to set this up.

Important: Even when a task is executed under a custom service account, the output artifacts produced by that task are still placed in the same artifact store shared with all other Oasis users.

Instructions​

In Shopify all cloud resource manipulations should be performed using Terraform via the https://github.com/Shopify/terraform-the-cloud/ repository. The repository is self-serve (your PRs can be approved by your team), but supported by the Shopify infra teams.

Per-team - Set up team's custom Service Account​

  1. Create a Google Cloud Service Account in some project. Example: sdp-stg-oasis-runner@sdp-stg-discovery-experience.iam.gserviceaccount.com. Example Example:

    resource "google_service_account" "sdp_stg_oasis_runner" {
      project      = module.data_domain_project.project_id
      account_id   = "sdp-stg-oasis-runner"
      display_name = "Oasis runner for Discovery Experience Staging"
      description  = "Shared service account for accessing cloud resources for Discovery Experience Staging"
    }

  2. Give that GSA any other desired permissions. Example: PR Section Example:

    resource "google_project_iam_member" "sdp_stg_oasis_runner_iam" {
      project = module.data_domain_project.project_id
      for_each = toset([
        "roles/bigquery.dataEditor",
        "roles/bigquery.jobUser",
        "roles/bigquery.readSessionUser",
        "roles/iam.serviceAccountUser",
        "roles/logging.logWriter",
        "roles/pubsub.publisher",
        "roles/run.invoker",
        "roles/run.viewer",
        "roles/secretmanager.secretAccessor"
      ])
      role   = each.key
      member = google_service_account.sdp_stg_oasis_runner.member
    }

  3. Set up google_service_account_iam_binding for the account, giving specific users permissions to use the team's GSA. See details in the next section: "Per-user - Allow a user to use the team's custom Service Account". link Example:

    # Grant Workload Identity permissions to individual users' Kubernetes Service Accounts
    # This allows those users to impersonate the sdp-stg-oasis-runner SA in their Oasis pipeline runs.
    resource "google_service_account_iam_binding" "grant_oasis_sa_with_user_list" {
      service_account_id = google_service_account.sdp_stg_oasis_runner.name
      role               = "roles/iam.workloadIdentityUser"

      members = [
        # For use with the new Oasis SA annotation: `cloud-pipelines.net/launchers/google/service_account: sdp-stg-oasis-runner@sdp-stg-discovery-experience.iam.gserviceaccount.com`
        # "principal://iam.googleapis.com/projects/430902663979/locations/global/workloadIdentityPools/shopify-skypilot-clusters.svc.id.goog/subject/ns/<team-service-account-at-team-project>/sa/<user-account-at-shopify-com>"
      ]
    }

  4. Give that GSA write access to Oasis' GCS bucket (prd-oasis-tmp). See this section

    Only this part requires a review from the Oasis team.

    resource "google_storage_bucket_iam_member" "oasis_intermediate_data_access_for_custom_service_accounts" {
        bucket   = google_storage_bucket.oasis_intermediate_data.name
        member   = each.key
        role     = "organizations/430199081660/roles/StorageUserOrg"  # Read/Write access
        for_each = toset(
          [
              "serviceAccount:sdp-stg-oasis-runner@sdp-stg-discovery-experience.iam.gserviceaccount.com",
          ]
        )
    }

Per-user - Allow a user to use the team's custom Service Account​

Now we need to allow some users to use the team's service account.

  1. Add a user-specific WorkloadIdentity principal to the team's GSA with the iam.workloadIdentityUser role. Example PR

Example:

# Grant Workload Identity permissions to individual users' Kubernetes Service Accounts
# This allows those users to impersonate the sdp-stg-oasis-runner SA in their Oasis pipeline runs.
resource "google_service_account_iam_binding" "grant_oasis_sa_with_user_list" {
  service_account_id = google_service_account.sdp_stg_oasis_runner.name
  role               = "roles/iam.workloadIdentityUser"

  members = [
    # For use with the new Oasis SA annotation: `cloud-pipelines.net/launchers/google/service_account: sdp-stg-oasis-runner@sdp-stg-discovery-experience.iam.gserviceaccount.com`
    "principal://iam.googleapis.com/projects/430902663979/locations/global/workloadIdentityPools/shopify-skypilot-clusters.svc.id.goog/subject/ns/sdp-stg-oasis-runner-at-sdp-stg-discovery-experience/sa/alexey-volkov-at-shopify-com",
  ]
}

The member principal format: "principal://iam.googleapis.com/projects/430902663979/locations/global/workloadIdentityPools/shopify-skypilot-clusters.svc.id.goog/subject/ns/<team-service-account-at-team-project>/sa/<user-account-at-shopify-com>"

The last sections of this member string (<team-service-account-at-team-project> and <user-account-at-shopify-com>) are Kubernetes resource names and must adhere to Kubernetes' resource name syntax. So, you'll need to convert the account name spelling in a specific way. Lowercase all letters, replace "@" with "-at-", replace all other symbols (e.g. ".") with "-". Also, the ".iam.gserviceaccount.com" SA suffix is removed.

The team's Google Service Account becomes a Kubernetes Namespace name:

From:   sdp-stg-oasis-runner@sdp-stg-discovery-experience.iam.gserviceaccount.com
To:     sdp-stg-oasis-runner-at-sdp-stg-discovery-experience

The user's Google User Account becomes a Kubernetes ServiceAccount name:

From:   alexey.volkov@shopify.com
To:     alexey-volkov-at-shopify-com

Per-task - Use custom service account in a pipeline.​

Custom service account should be set for each pipeline task that requires it.

  1. When editing a pipeline, select a pipeline Task node, navigate to the "Annotations" tab in the right side panel. Add a new annotation with name cloud-pipelines.net/launchers/google/service_account and specify the team's service account as value (e.g. sdp-stg-oasis-runner@sdp-stg-discovery-experience.iam.gserviceaccount.com). In the future, there will be a dedicate UX input for this.

Now, when the user submits the pipeline, the task annotated with teh custom SA will be executed under that SA.

Design and implementation details.​

There are multiple design problems that need to be solved while designing a solution for custom service accounts.

  • How can the user specify the team SA that they want to use?
  • How can the system know whether the user can use the team's SA? Can this access be revoked?
  • What happens when some other user clones the pipeline that has some tasks annotated with team SA?

Implementation:​

Oasis pipeline tasks are executed as Kubernetes Pods. Kubernetes has concept of ServiceAccounts, but this is a lightweight abstraction that only exists inside the Kubernetes Cluster and is pretty disconnected from the world outside the cluster. However, Google Kubernetes Engine support a feature called Workload Identity which allows to bind a KSA (Kubernetes Service Account) to GSA (Google Service Account). This binding needs to be established on both sides - Kubernetes (via iam.gke.io/gcp-service-account annotation on ServiceAccount) and Google Cloud IAM (via the roles/iam.workloadIdentityUser permission).

When launching a pipeline task for execution the Oasis backend does the following:

By default, use the default oasis-service-account KSA that's bound to oasis-service-account@shopify-skypilot-clusters.iam.gserviceaccount.com GSA. However, if the cloud-pipelines.net/launchers/google/service_account annotation is specified, then take the custom Google Service Account, the Google User Account and:

  1. If needed, create the Kubernetes Namespace corresponding to the team's Google Service Account: Example: sdp-stg-oasis-runner-at-sdp-stg-discovery-experience
  2. If needed, create the user Kubernetes ServiceAccount in the Kubernetes Namespace. Example: alexey-volkov-at-shopify-com
  3. Bind the user KSA to team GSA via the iam.gke.io/gcp-service-account annotation.
  4. Create Kubernets Pod with namespace: sdp-stg-oasis-runner-at-sdp-stg-discovery-experience and serviceAccountName: alexey-volkov-at-shopify-com
